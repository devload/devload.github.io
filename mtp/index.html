Chapter 0

이 기술이 뭔데?

MTP (Multi-Token Prediction)는 AI가 글을 쓸 때, 한 단어씩 만드는 대신 여러 단어를 한꺼번에 예측해서 속도를 끌어올리는 기술이에요. 원래는 2024년 Meta가 논문으로 처음 제안했고, 2026년 5월에 Google이 Gemma 4용 MTP Drafter를 공개하면서 화제가 됐어요.

🎯 한 줄 요약

작은 모델(Drafter)이 다음 N개 단어를 미리 추측해서 적어두면, 큰 모델은 그걸 한 번에 검사하기만 하면 돼요. 검사는 큰 모델이 직접 하니까 품질은 그대로, 속도는 최대 3배!

Chapter 1

도대체 뭐가 느린 거야?

ChatGPT나 Gemini 같은 AI는 글을 쓸 때 단어(토큰)를 한 번에 하나씩 만들어요. 다음 단어를 만들려면 지금까지 만든 모든 단어를 다시 모델에 넣고, 또 한 번 거대한 계산을 해야 해요. 이걸 "자동회귀 생성(Autoregressive Generation)"이라고 불러요.

비유 1 · 받아쓰기

선생님이 한 글자씩 부르고, 학생이 받아쓴 다음 또 한 글자 들으러 가는 상황이에요. 100글자 받아쓰려면 100번 왔다 갔다. 왕복하는 시간이 글자 쓰는 시간보다 더 길어요.

AI도 마찬가지예요. 모델 가중치를 GPU 메모리에서 읽어오는 시간이 실제 계산보다 훨씬 길어요. 그래서 한 번 읽어올 때 한 글자만 만드는 게 너무 아깝죠.

왜 이게 중요한가요?

💸 비용

  • 같은 답을 만들면서 전기료 1/3
  • 같은 GPU로 3배 많은 사용자 처리
  • 데이터센터 운영비 절감

⚡ 사용자 경험

  • 답변 대기 시간이 1/3
  • 스트리밍 응답이 더 매끄러움
  • 로컬 노트북에서도 큰 모델 실행 가능
Chapter 2

지금까지는 어떻게 빠르게 했어?

AI를 빠르게 만드는 방법은 여러 가지가 있었어요. 하지만 대부분 "품질을 조금 포기하고 속도를 얻는" 거래였죠.

방법 비유 속도 품질
양자화
(Quantization)		 📐 정밀한 자 대신 거친 자로 재기 ↑ 빨라짐 ↓ 살짝 저하
가지치기
(Pruning)		 ✂️ 별로 안 쓰는 가지를 잘라냄 ↑ 빨라짐 ↓ 저하 위험
증류
(Distillation)		 👨‍🏫 작은 모델이 큰 모델 따라하기 ↑↑ 매우 빠름 ↓ 한계 있음
🌟 MTP
(이 글의 주인공)		 🤝 작은 모델 추측 + 큰 모델 검증 ↑↑ 최대 3배 ✓ 완전 동일
MTP가 특별한 이유

다른 방법들은 모두 "품질을 조금 포기하고 속도를 얻는" 거래였어요. 하지만 MTP는 품질을 전혀 포기하지 않으면서 속도만 얻습니다. 어떻게 그게 가능할까요? 다음 챕터에서 살펴봐요.

Chapter 3

MTP는 어떻게 작동해?

MTP의 핵심은 드래프터(Drafter)와 검증자(Verifier)의 환상의 콤비예요. 작은 모델이 추측하고, 큰 모델이 검증하는 방식이죠.

한 장면으로 이해하기

📝 1단계: 추측
작은 Drafter가 다음 4개 단어를 빠르게 추측
↓
🔍 2단계: 검증
큰 Target 모델이 4개를 한 번의 forward pass로 병렬 검증
↓
✅ 3단계: 채택
맞은 토큰은 한꺼번에 채택 + 보너스 1개 더!
비유 2 · 받아쓰기 짝꿍

이번에는 옆에 짝꿍이 생겼어요!

  • 짝꿍 (Drafter): 선생님 입모양만 보고 "다음 4글자는 아마 '안녕하세요'일 거야!" 하고 미리 적어둠
  • 나 (Target Model): 짝꿍이 쓴 4글자를 보고 "맞네!" 한 번에 검사
  • ✅ 맞으면 → 4글자가 한 번에 완성, 다음 1글자도 보너스
  • ❌ 틀린 곳이 있으면 → 그 지점부터 내가 직접 다시 씀

4가지 핵심 기술

1️⃣ Draft Model

미리 추측하는 작은 모델

  • 본 모델보다 30~100배 작음
  • Gemma 4 기준 약 2~3억 파라미터
  • 빠르지만 가끔 틀려도 OK

2️⃣ Speculative Verification

병렬 검증

  • N개 토큰을 한 번에 검사
  • GPU 병렬성을 그대로 활용
  • 1개 만드는 시간 = N개 검증 시간

3️⃣ KV 캐시 공유

Drafter가 본 모델에 기생

  • 본 모델의 activation을 빌려 씀
  • 같은 컨텍스트 재계산 없음
  • Drafter의 추가 비용이 거의 0

4️⃣ +1 보너스 토큰

덤으로 받는 한 개

  • 검증 중에 다음 토큰 자동 생성
  • Drafter 4개 → 최대 5개 출력
  • 한 번 돌리는 시간에 5개!
중요 · 품질은 왜 그대로일까?

검증은 본 모델이 직접 하기 때문이에요. Drafter가 어떤 추측을 하든, 본 모델이 동의하지 않으면 채택되지 않아요. 그래서 최종 출력은 본 모델 혼자 한 것과 100% 동일합니다. 단지 본 모델을 돌리는 횟수만 줄어든 거예요.

Chapter 4

Gemma 4는 뭐가 특별해?

MTP 자체는 일반적인 기법이에요. 하지만 Google이 Gemma 4에 맞춰 만든 MTP Drafter는 몇 가지 특별한 점이 있어요.

🧠

본 모델의 뇌를 빌려 씀

Drafter가 본 모델의 internal activation을 그대로 활용. 따로 계산할 필요 없음.

🗂️

KV 캐시 공유

같은 컨텍스트를 두 번 인코딩하지 않음. 메모리도 절약.

💻

코딩 태스크에 특화

코드는 패턴이 정형화돼서 추측 적중률이 높음. 그래서 가속 효과가 가장 큼.

어디서 받을 수 있어?

Apache 2.0 라이선스로 무료 공개됐어요. 대부분의 인기 런타임에서 지원해요.

🦙 Ollama 로컬 실행 ✓ Gemma 4 한정

Ollama로 써보기

가장 간단한 방법은 Ollama예요. Modelfile에 DRAFT 한 줄만 추가하면 끝나요.

# Modelfile 만들기 FROM gemma4:9b DRAFT gemma4-9b-mtp # 빌드 후 실행 $ ollama create my-fast-gemma -f Modelfile $ ollama run my-fast-gemma "FastAPI 코드 짜줘"

Chapter 6

정리하면?

MTP는 단순히 "AI가 좀 빨라지는" 기술이 아니에요. AI 비용 구조 자체를 바꾸는 기법이에요.

🚀 MTP가 열어주는 미래

  • 전기료 1/3: 같은 답을 더 적은 에너지로
  • 로컬 AI 시대: 노트북에서도 큰 모델이 부드럽게
  • 실시간 응답: 의료 진단, 동시통역 등 즉시성 필요한 영역
  • 탄소 발자국 감소: 데이터센터 전력 소비 절감

다른 모델들도 따라할까?

이미 따라하고 있어요. DeepSeek V3는 학습 단계부터 MTP를 적용했고, Qwen 3.6도 MTP 지원을 추가했어요. 앞으로는 거의 모든 LLM이 MTP나 비슷한 Speculative Decoding 기법을 기본 탑재할 거예요.

자주 묻는 질문 · Claude Code 같은 클라우드 AI는?

사용자가 직접 MTP를 켜고 끌 수는 없어요. 클라우드 서비스는 제공자(Anthropic, OpenAI 등)가 서버 안에서 알아서 최적화하거든요.

하지만 안심하세요. 업계 표준이 된 만큼, 여러분이 쓰는 클라우드 AI도 비슷한 가속 기법으로 이미 빨라져 있을 가능성이 높아요.

"MTP는 AI한테 예지력 있는 짝꿍을 붙여준 기술 — 미리 답을 추측해두니까 본 모델은 검사만 하면 돼서 3배 빠르고, 답은 똑같다!"

용어사전

MTP (Multi-Token Prediction)
한 번에 여러 토큰을 예측하는 기법. 이 글의 주인공.
토큰 (Token)
AI가 글을 읽고 쓰는 최소 단위. 대략 단어 하나~두 개.
자동회귀 (Autoregressive)
단어를 한 개씩 순서대로 생성하는 방식. 정확하지만 느림.
Drafter (드래프터)
다음 토큰들을 미리 추측하는 작은 모델. "비서" 역할.
Target Model
실제 답을 만들어야 하는 큰 모델. "사장님" 역할.
Speculative Decoding
"추측 디코딩". MTP의 학술적 이름. Drafter가 추측하고 Target이 검증하는 방식.
KV 캐시
AI가 이미 읽은 내용을 임시 저장하는 공간. 매번 재계산을 피함.
Forward Pass
모델 한 번 통과시키는 계산. 비용이 커서 횟수 줄이는 게 핵심.
Activation
모델 내부의 중간 계산 결과. Drafter가 빌려쓰면 자기 계산을 줄일 수 있음.
Gemma 4
Google이 만든 오픈소스 LLM. Apache 2.0 라이선스.
MoE (Mixture of Experts)
여러 "전문가" 모델을 합쳐 만든 모델. 26B MoE는 26B 파라미터 MoE 모델.
vLLM / SGLang
LLM을 빠르게 서빙하는 추론 엔진들. MTP를 잘 지원.
Ollama
로컬에서 LLM을 쉽게 실행하는 도구. ollama run 한 줄로 실행.
Apache 2.0
오픈소스 라이선스. 상업적 사용 포함 자유롭게 사용 가능.
