AI Agent Memory
다섯 진영을 가르는 진짜 기준

Mem0 · Letta · Memoir · Obsidian + MCP · Zep — 같은 'AI 메모리' 시장처럼 보이지만, 다섯 도구는 사실 서로 다른 종류의 문제를 풀고 있다.

📅 2026년 5월 ⏱ 약 15분 🔍 5-way 분석 + 3개 1:1 비교
왜 이 글이 존재하는가

같은 문제, 다섯 가지 풀이

"AI에게 어떻게 기억을 줄까"는 아직 업계 합의가 없는 질문이다. CLAUDE.md 같은 단일 파일, 벡터 DB의 임베딩, 매 세션 클린 슬레이트 — 모두 부분적 해법이고 각자 한계가 있다.

프로덕션 에이전트가 결국 부딪히는 공통적인 세 가지 문제가 있다:

🧪

Context Contamination

한 작업/브랜치에서 배운 패턴이 무관한 작업에 새어 나옴. 컨텍스트 격리가 어려움.

💸

Token Rent

메모리 한 줄 수정해도 prefix cache가 통째로 무효화 → 매 턴 전체 재처리.

🌀

Memory Drift

잘못된 학습이 누적되어도 추적·되돌리기가 어려움. 메모리 디버깅 도구 부재.

다섯 도구는 이 세 문제를 서로 다른 위치에서 공격한다. "어느 게 더 낫다"가 아니라 당신 문제와 같은 위치에서 싸우는 도구가 무엇이냐가 핵심이다.

📎 위 세 문제의 분류는 Memoir 진영에서 처음 정리한 프레임이지만, 다섯 도구 모두 이 중 일부를 의식적으로 풀려고 한다 — 즉 비교 축으로는 유효하다.

MAP한눈에 보는 포지셔닝

다섯 도구가 어떻게 갈리는지 두 축으로 본다 — "메모리 변경을 어떻게 다루나"(진실 모델)와 "누가 메모리를 결정하나"(관리 주체). 같은 도구라도 두 축에서 자리가 다르다.

Agent Runtime
Letta에이전트 런타임 (구 MemGPT)
⭐ 18k+ stars

"LLM as Operating System" — 에이전트가 자기 메모리를 직접 관리하도록 만든 풀스택 런타임.

🧩 저장 구조 — OS 비유 3계층
Core
RAM · 항상 in-context
Recall
디스크 캐시 · 대화 검색
Archival
콜드 스토리지 · 벡터
⚙️ 작동 흐름
# 에이전트의 reasoning loop 안에서 # 에이전트가 판단해서 직접 호출: agent.core_memory_append("Sarah prefers tabs") agent.archival_memory_insert(...) # Core 꽉 차면 agent.archival_memory_search(...) # 옛 일 떠올릴 때 # 무엇을 어디에 저장할지 에이전트가 스스로 결정 — Mem0와 정반대
📍 포지션
관리 주체에이전트 자신 (tool call로 self-edit)
진실 모델에이전트의 판단이 진실 (자율적이지만 책임도 큼)
메모리 계층Core / Recall / Archival (OS의 RAM/캐시/콜드)
락인높음 — 에이전트 reasoning loop 전체를 Letta가 소유
⚖️ 강·약
STRENGTHS
  • 장기 호라이즌 일관성에 강함
  • LongMemEval 83.2% (오픈소스 상위권)
  • 2026-01 Conversations API — 다중 에이전트 공유
  • MemGPT 논문 기반, 학술적 기반 탄탄
WEAKNESSES
  • 런타임 락인 — 이미 짠 loop를 버려야
  • 관리형 클라우드 아직 개발 중
  • 셋업·인프라 부담 큼
  • 이력·브랜치 1급 지원 없음
🎯 이런 사람에게
며칠~몇 주 동안 자율로 일하는 에이전트가 핵심 제품일 때. 에이전트가 스스로 무엇을 기억할지 결정해야 하는 시나리오.

⭐ ~400 (alpha)

"git-like"가 비유가 아니다. literally git이다. ProllyTree(Merkle B-tree)가 실제 .git 디렉토리를 만든다.

🧩 저장 구조 — 진짜 git commit tree
●7f3a2b1mainadd preferences.coding.style
│
●9b4c8d2add profile.identity.name
│ ╲
│ ●3c8d1e0experimenttry new framework
│ ╱
●2a9f6c4initial commit
⚙️ 작동 흐름
$ memoir new my-store # .git 디렉토리 실제 생성 $ memoir remember "Sarah prefers tabs" └─ LLM이 시맨틱 경로 분류: preferences.coding.style └─ ProllyTree에 저장 → 자동 git commit $ git log --oneline # 진짜 git 명령어 동작 $ git checkout 9b4c8d2 # 옛 메모리 상태로 시간 여행 $ git branch experiment # 평행 우주 메모리
📍 포지션
관리 주체LLM 분류기(자동 경로) + 사람(commit/branch 결정)

자료구조ProllyTree (Dolt 영감) + 실제 git 저장소 — SHA-256 무결성
락인중간 — git 호환이라 데이터는 빼낼 수 있음
⚖️ 강·약

  • 코드 git 브랜치와 페어링 가능
  • 경로 기반 핀포인트 조회 (cache 친화)

WEAKNESSES
  • 알파 단계, 생태계 작음 (~400 stars)
  • 벤치마크 공개 없음

🎯 이런 사람에게

Temporal Knowledge Graph
Zep (Graphiti)시간 인지 지식 그래프 메모리 서버
⭐ Graphiti 14k+ (8개월)

사실(fact)마다 "언제부터 참이었고 언제 무효화됐는지" 시간 윈도를 단다. 갱신 시 옛 사실을 지우지 않고 invalidate한다.

🧩 저장 구조 — fact마다 validity window
(Sarah) — works_at → (Acme) 2024-03 → 2025-08
(Sarah) — works_at → (Globex) 2025-08 → ∞
(Sarah) — prefers → (tabs) 2024-09 → ∞
⚙️ 작동 흐름
# 1. 데이터 ingestion (메시지·문서·JSON) zep.add(messages=[...]) └─ Graphiti가 entity/relationship/fact 추출 └─ 각 fact에 validity window 부여 # 2. 상태가 변하면 옛 fact를 invalidate (삭제 X) # 3. 검색은 hybrid (semantic + BM25 + graph), LLM 호출 없음 zep.search("where does Sarah work?") # 지금 zep.search("... as of 2025-01") # 시간 여행
📍 포지션
관리 주체시스템 (async 추출 — Mem0와 비슷하지만 그래프 기반)
진실 모델시간축 위의 fact — "T시점에 무엇이 참이었나" 답함
자료구조Temporal Knowledge Graph (valid_from / valid_to)
락인중간 — Zep 서버 의존, Graphiti만 따로 self-host 가능
⚖️ 강·약
STRENGTHS
  • 시간 추론에 압도적 강함 (LongMemEval 71.2%)
  • 사용자 상태 변화 추적 — 이력 보존
  • 검색 시 LLM 호출 없음, P95 300ms
  • Graphiti 8개월에 14k stars
WEAKNESSES
  • 브랜치 개념 없음 (병렬 컨텍스트 격리 불가)
  • 서버 인프라 운영 부담
  • 그래프 스키마 설계 학습 곡선
  • Mem0보다 무거운 도입
🎯 이런 사람에게
엔터프라이즈 컨텍스트에서 사용자/엔티티 상태가 시간에 따라 변하고, 그 이력이 비즈니스적으로 중요할 때. CRM·의료·컴플라이언스처럼 "언제 무엇이 참이었는지"가 답할 가치가 있는 영역.

PART 2같은 입력, 다섯 가지 반응

추상적 비교는 와닿지 않는다. 같은 사건이 들어왔을 때 다섯 도구가 실제로 무엇을 하는지 나란히 보면 차이가 즉시 드러난다.

🎬 시나리오 ①   2025년 8월. 사용자가 알린다: "나 Acme에서 Globex로 이직했어". 메모리에는 (Sarah, works_at, Acme, since 2024-03)가 이미 저장되어 있다.
Mem0
옛 fact를 덮어쓰기. 메모리에는 works_at = Globex만 남음. "2024년에 어디 일했어?"는 답 못함.
Letta
에이전트가 reasoning loop에서 판단 — core_memory_replace() 또는 옛 정보는 archival로 옮기고 새 정보를 core에. 결과는 에이전트 의사결정에 달림.
Memoir
새 commit 생성: profile.work.company 경로에 Globex 저장. 옛 commit은 그대로 — git checkout으로 2025-07 시점 메모리로 시간 여행 가능.
Obsidian + MCP
에이전트가 profile.md 파일을 열어 "Acme" 줄을 "Globex"로 수정. 사람도 같은 파일을 봄. git plugin 켜져 있으면 변경이 기록됨.
Zep
옛 fact를 invalidate (삭제 X): valid_to=2025-08 설정. 새 fact 추가: (Sarah, works_at, Globex, valid_from=2025-08). 두 fact 모두 그래프에 남음.
🎬 시나리오 ②   코딩 에이전트가 main 브랜치에서 "이 프로젝트는 React + Redux"라고 학습. 그 다음 git checkout feature/migrate-to-zustand로 전환했다.
Mem0
메모리는 git을 모른다. 여전히 "Redux 패턴 추천" — feature 브랜치에 Redux 코드를 끼워 넣어 컨텍스트 오염 발생.
Letta
에이전트가 직접 인지하지 않으면 동일 문제. 다만 reasoning loop에서 "지금 브랜치가 다르네"를 알아챌 수 있는 자율성은 있음.
Memoir
메모리 자체에 브랜치가 있음. git checkout feature/... 시 메모리도 함께 전환 — Redux 컨텍스트는 main 브랜치에만 머무름. 본 시나리오의 정확한 해결책.
Obsidian + MCP
vault 자체가 git repo라면 브랜치별로 다른 노트 가능. 하지만 일반적으로 한 vault만 보고 있어 같은 문제 발생 가능.
Zep
브랜치 개념 없음. 시간축에는 강하지만 "병렬 컨텍스트 격리"는 약점.
🎬 시나리오 ③   어제 세션에서 에이전트가 잘못된 정보를 학습했다: "이 함수는 thread-safe함" (사실 아님). 오늘 그 사실을 알아챘다. 어떻게 되돌리나?
Mem0
새 정보로 update — 옛 잘못된 fact는 덮여 사라짐. 다만 "어느 세션에서 어떻게 들어왔는지" 추적은 어려움.
Letta
에이전트에게 알려주면 reasoning으로 core/archival을 수정. 다만 잘못된 정보로 이미 다른 결정이 내려졌으면 그 영향까지는 못 되돌림.
Memoir
memoir blame <경로>로 언제 들어왔는지 식별 → memoir checkout <그 전 commit> 또는 git revert. 코드 디버깅처럼 메모리도 디버깅.
Obsidian + MCP
git plugin 있으면 git log로 추적 가능. 마크다운 한 줄 단위까지 blame.
Zep
잘못된 fact를 invalidate하고 새 fact 추가. validity 메타데이터에 출처 기록되어 있어 "언제 어떻게 들어왔는지"는 추적 가능.

실무 결정 트리

"어느 게 더 좋아?"가 아니라 "내 문제는 어디서 싸우는가?"가 맞는 질문이다. 단, 아래 매핑은 대표적 케이스일 뿐 — 실제 프로덕션에서는 둘을 결합하거나(예: Letta + Zep 백엔드), 어느 것도 완벽히 맞지 않는 회색 영역이 흔하다.

컨슈머 챗봇/어시스턴트에 "사용자를 기억" 기능을 빠르게 붙이고 싶다
→ Mem0
에이전트가 며칠~몇 주 동안 자율로 일해야 하고, 장기 일관성이 곧 제품
→ Letta
코딩 에이전트가 git checkout마다 컨텍스트가 따라 갈아끼워져야 한다
→ Memoir
사람도 같이 보는 second brain, 이미 옵시디언 vault에 자산이 쌓여 있다
→ Obsidian + MCP
사용자/엔티티 상태가 시간에 따라 변하고 그 이력이 비즈니스적으로 중요하다
→ Zep

PART 51:1 Deep Dives — 가장 흥미로운 3쌍

전체 비교가 아니라 대비가 가장 선명한 두 도구씩 묶어서, 각자의 설계 철학이 정반대로 갈리는 지점을 본다. 세 페어는 각각 다른 축의 대비를 보여준다 — 시간성·관리 주체·변경 처리.

① Memoir ↔ Zep — 시간성의 두 모델

다섯 중 과거를 1급으로 다루는 두 도구인데, 모델이 정반대다. Memoir는 코드 브랜치처럼 병렬성을, Zep은 DB temporal model처럼 시간성을 모델링한다.

Memoir — 이산 스냅샷 (git)
commit 1: profile.skill.python
          = "intermediate"
commit 2: profile.skill.python
          = "expert"

$ git checkout commit-1
→ 옛 상태 전체 복원

$ git checkout -b experiment
→ 평행 우주 메모리

코드 브랜치마다 다른 메모리. 실험 컨텍스트가 프로덕션 메모리를 오염시키지 않음.

Zep — 연속 시간축 (DB temporal)
Fact: (Sarah, skill, python,
       intermediate)
  valid_from=2024-03
  valid_to=2025-08

Fact: (Sarah, skill, python,
       expert)
  valid_from=2025-08
  valid_to=∞

→ "2024-05의 Sarah 실력?"
   → intermediate

시간이 흐르며 변하는 단일 진실. 브랜치 개념 없음. "언제 무엇이 참이었나"를 정확히 답함.

핵심: 당신 도메인에서 "버전"이 의미 있나(Memoir), "시점"이 의미 있나(Zep)? 코드·문서·실험이라면 전자, CRM·의료·컴플라이언스라면 후자.

② Mem0 ↔ Letta — 관리 주체의 정반대

둘 다 "에이전트의 메모리 관리"를 풀지만 누가 결정하느냐에서 정반대 진영. Mem0는 시스템이 알아서, Letta는 에이전트가 직접.

Mem0 — 시스템이 결정 (passive)
mem0.add(conversation)
   ↓
[LLM 추출 파이프라인]
   ↓ 자동으로:
- 어떤 fact를 뽑을지
- 어디에 저장할지
- 중복은 어떻게 처리할지
- 모순은 어떻게 update할지
모두 시스템이 결정.

개발자: input 던지면 끝.

"기억을 시스템에 위임" — 라이브러리처럼 쓰면 됨. 락인 낮음. 다만 디버깅·통제는 약함.

Letta — 에이전트가 결정 (self-edit)
# reasoning loop 안에서
agent.core_memory_append(...)
agent.archival_memory_insert(...)
agent.archival_memory_search(...)

# 에이전트가 판단:
- 지금 이 정보가 중요한가?
- core에 둘까 archival로 갈까?
- 옛 정보를 지울까 옮길까?
모두 에이전트가 결정.

"기억을 에이전트에게 위임" — 자율성·일관성 강함. 다만 reasoning 비용·loop 락인 큼.

핵심: 메모리가 외부 서비스이길 원하는가(Mem0), 에이전트의 자율 영역이길 원하는가(Letta)? "그냥 기억해줘"면 전자, "스스로 판단해야 해"면 후자.

③ Mem0 ↔ Zep — 같은 자동화, 정반대 변경 처리

둘 다 시스템 자동 추출 진영이지만, 모순된 정보가 들어왔을 때 옛 사실을 어떻게 다루냐에서 정반대. Mem0는 지우고, Zep은 무효화한다.

Mem0 — 덮어쓰기 (overwrite)
# 옛 상태:
(Sarah, works_at, Acme)

# 새 입력: "이직했어 → Globex"

mem0.add(new_message)
   ↓ LLM이 모순 감지
   ↓ 옛 fact를 update
(Sarah, works_at, Globex)
# Acme은 사라짐

# "2024년 Sarah 직장?"
→ ❌ 답 못함

지금 사실이 곧 진실. 가볍고 빠르지만 시간 추론 불가.

Zep — 무효화 (invalidate)
# 옛 상태:
(Sarah, works_at, Acme,
 valid_from=2024-03)

# 새 입력: "이직했어 → Globex"

zep.add(new_message)
   ↓ Graphiti가 모순 감지
   ↓ 옛 fact에 valid_to 설정
(Sarah, works_at, Acme,
 valid_from=2024-03,
 valid_to=2025-08)  ← invalid

(Sarah, works_at, Globex,
 valid_from=2025-08)  ← valid

# "2024년 Sarah 직장?"
→ ✅ Acme

옛 사실도 시간축 위에 남음. 무겁지만 시간 추론 가능.

핵심: "지금 사실"만 알면 되나(Mem0), "언제 무엇이 참이었나"도 알아야 하나(Zep)? 챗봇이면 전자로 충분하지만, 엔터프라이즈 데이터라면 후자가 필요.

END요약

다섯 도구는 같은 "AI 메모리" 시장에서 경쟁하는 것처럼 보이지만, 실제로는 서로 다른 종류의 문제를 푼다.

  • Mem0 — "기억을 자동으로 만들고 자동으로 갱신해줘"
  • Letta — "에이전트가 자기 머릿속을 직접 관리하게 해줘"
  • Memoir — "메모리도 코드처럼 branch / commit / blame 되게 해줘"
  • Obsidian + MCP — "내가 보는 노트와 AI 메모리가 같은 파일이게 해줘"
  • Zep — "사실이 시간에 따라 어떻게 변했는지 추적해줘"

도구를 고를 때 "어느 게 더 좋아?"는 잘못된 질문이다.
정확한 질문은 "내 문제는 어느 시간 모델·관리 주체에 속하는가?"이다.
