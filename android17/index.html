Android 17이 앱에 "메모리 상한"을 씌운다
App Memory Limit 완전 정리

Android 17 Beta 4에서 조용히, 그러나 묵직하게 등장한 변화 하나 — OS가 앱별로 RAM 사용 상한을 강제하기 시작한다. 왜 이런 게 나왔고, 어떻게 동작하며, 우리는 뭘 준비해야 하는가.

지난 10년간 Android는 "원하는 만큼 써, 대신 운 나쁘면 누가 죽을지 몰라" 였다.
Android 17부터는 "네 몫만큼 써. 넘치면 네가 죽는다." 로 바뀐다.

TL;DR

  • Android 17부터 기기 총 RAM에 기반한 앱별 메모리 상한이 도입된다.
  • 상한 초과 앱은 종료되며, ApplicationExitInfo.getDescription()에 "MemoryLimiter"가 찍힌다.
  • 초기에는 보수적으로 적용 — 심한 누수 앱만 우선 타겟팅.
  • 철학은 "LMK의 복불복 종료를 끝내고, 원인 앱에 직접 책임을 묻는 구조"로 전환.
  • 대응: ProfilingManager의 TRIGGER_TYPE_ANOMALY 자동 힙덤프, LeakCanary(Android Studio Panda 통합), ApplicationExitInfo 모니터링.

1. 배경: 왜 지금 "메모리 상한"인가

1-1. LMK(Low Memory Killer)의 오래된 한계

Android는 지난 10년 넘게 메모리가 부족하면 Low Memory Killer가 뒤에서 앱을 치워왔다. 동작 원리는 간단하다.

  1. 시스템이 메모리 압박을 감지한다.
  2. 각 프로세스의 oom_score_adj(중요도 점수)를 확인한다.
  3. 점수 높은 프로세스부터 차례로 SIGKILL 을 보낸다.

문제는 이 방식이 사후 대응이고, 피해자를 가해자와 무관하게 선정한다는 점이다.

상황기존 LMK의 행동
누수 심한 앱 A가 RAM 1.5 GB 점유A는 포그라운드라 살아남음
무고한 앱 B가 백그라운드에 있음B가 먼저 죽음 (우선순위 낮음)
사용자는 B를 다시 열 때 상태 유실원인 제공자 A는 멀쩡

즉 "메모리를 잘못 쓴 앱이 아닌, 우선순위만 낮은 앱이 벌을 받는" 구조. 이 비결정성이 수많은 jank와 콜드 스타트, 배터리 소모의 뿌리였다.

1-2. 앱 생태계가 점점 뚱뚱해지고 있다

실제 필드 데이터에서 구글이 관찰한 것은 꽤 노골적이다.

  • 일부 앱이 수백 MB~1 GB+ 를 기본으로 점유
  • 메모리 누수가 방치된 채로 스토어에 배포됨
  • 한 앱의 누수가 전체 기기의 배터리·발열·반응성을 망가뜨림
  • 사용자는 원인을 알 수 없고 "폰이 느려졌다"고만 느낌

앱은 많아지고 기능은 복잡해지는데, RAM 효율성에 대한 책임 구조가 없었다. 요청하면 주는 구조였고, 사고가 나면 LMK가 무작위로 솎아냈다.

1-3. 폼 팩터 다양화 — 하나의 정책으로는 안 된다

Android가 올라타는 기기 폭이 커졌다.

저가 스마트폰

3~4 GB RAM · 인도·동남아·남미 등 여전히 주력 시장. 메모리 압박이 즉시 UX로 드러남.

플래그십·폴더블

12~16 GB RAM · 멀티윈도우·멀티앱 시나리오가 메모리를 더 태움.

태블릿·차량·XR

폼 팩터별로 허용 예산이 전혀 다름. 단일 정책은 불가능.

Wear OS

극도로 제한된 자원. 한 앱의 누수가 워치 전체를 멈춤.

같은 앱이라도 4 GB 기기와 16 GB 기기에서 허용 가능한 예산은 완전히 다르다. "총 RAM 기반의 동적 상한" 이라는 설계가 이제서야 나온 이유다.

1-4. iOS Jetsam 모델로의 수렴

iOS는 수년간 Jetsam이라는 유사 개념을 운영해 왔다. 각 앱은 기기 RAM에 따른 예산을 받고, 초과 시 0xDEAD10CC 같은 식별 가능한 코드로 종료된다. 개발자는 Xcode/Organizer에서 원인을 정확히 확인할 수 있다.

Android도 결국 "앱별 예산제" 로 수렴하는 것. App Memory Limit이 그 신호탄이다.

2-3. 기존 LMK와의 철학적 차이

Before

LMK 중심

  • 시스템 메모리 부족 이후에 작동
  • oom_score_adj 우선순위로 제물 선정
  • 무고한 백그라운드 앱이 피해자
  • 원인 앱은 멀쩡히 계속 동작
  • 개발자는 원인을 역추적하기 어려움
After

App Memory Limit

  • 앱별 상한 초과 즉시 작동
  • 상한을 넘긴 원인 앱 본인을 종료
  • 무고한 앱은 살아남음
  • ApplicationExitInfo에 명확한 라벨
  • 개발자는 결정적으로 원인 식별

3. 개발자 체크리스트

3-1. ApplicationExitInfo로 종료 사유 집계

운영 단계에서 가장 먼저 심어야 하는 코드다. 빌드별로 "MemoryLimiter로 얼마나 죽는가"를 대시보드로 보는 것이 시작점.

val am = getSystemService(ActivityManager::class.java)
val exits = am.getHistoricalProcessExitReasons(packageName, 0, 20)

for (info in exits) {
    val killedByMemoryLimit = info.description?.contains("MemoryLimiter") == true
    if (killedByMemoryLimit) {
        reportMemoryLimiterExit(
            timestamp = info.timestamp,
            pss = info.pss,
            rss = info.rss,
            importance = info.importance,
            processName = info.processName
        )
    }
}
팁: reason 상수(REASON_EXCESSIVE_RESOURCE_USAGE 등)만으로는 LMK 종료와 구분이 안 될 수 있다. description의 "MemoryLimiter" 문자열이 Android 17 상한 정책의 결정적 식별자다.

3-2. ProfilingManager로 이상 시점 힙덤프 자동 수집

Android 15부터 도입된 ProfilingManager가 이번 정책과 잘 맞물린다. TRIGGER_TYPE_ANOMALY를 등록하면 시스템이 이상을 감지한 순간 프로파일이 자동으로 수집된다.

val profilingManager = getSystemService(ProfilingManager::class.java)

val triggers = listOf(
    ProfilingTrigger.Builder(ProfilingTrigger.TRIGGER_TYPE_ANOMALY).build()
)

val executor = Executors.newSingleThreadExecutor()
val callback = Consumer<ProfilingResult> { result ->
    if (result.errorCode == ProfilingResult.ERROR_NONE) {
        uploadHeapDump(result.resultFilePath) // 서버로 전송해 분석
    }
}

profilingManager.registerForAllProfilingResults(executor, callback)
profilingManager.addProfilingTriggers(triggers)

"문제가 터졌을 때 누군가가 덤프를 뜨기를" 기다릴 필요가 없어진다. 이상이 감지된 그 시점의 힙이 자동으로 잡힌다.

3-3. LeakCanary × Android Studio Panda

Android Studio Panda 버전부터 LeakCanary가 Profiler에 정식 통합된다.

  • 별도 의존성 추가 없이 IDE 태스크로 실행 가능
  • 누수 참조 체인을 소스 코드와 직접 연결해 보여줌
  • 기존 LeakCanary 라이브러리를 쓰는 팀은 계속 병행 사용 가능

현실적으로 대부분의 "MemoryLimiter 종료"는 누수가 원인이다. 누수 제거 = 이번 정책에 대한 가장 확실한 방어.

3-4. 평상시 메모리 예산 관리

OS가 강제하기 전에, 팀 자체의 메모리 예산을 운영하는 것이 좋다.

베이스라인 설정

주요 화면별 idle PSS/RSS를 측정하고 CI에 기록. Macrobenchmark + Baseline Profile로 회귀 감지.

Bitmap · Cache 튜닝

LruCache 크기 상한 점검, Bitmap.Config 검토, Glide/Coil 메모리 캐시 정책 조정.

누수 상습범 점검

Context/View 누수, static 컬렉션, Handler·리스너 해제 누락, 코루틴 스코프 관리.

대용량 객체 정리

오래 살아있는 싱글톤에 ML 모델·큰 JSON·Bitmap을 얹고 있지 않은지 점검.

4. 철학: "Deterministic Android"

이번 변경에서 구글이 유독 강조하는 단어는 deterministic(결정적) 이다.

지금까지

메모리 먹는 앱 때문에 무고한 앱과 OS가 욕을 먹던 시대. "왜 죽었는지 모르겠다"가 기본값.

앞으로

원인이 정확히 죽고, 이유가 명확히 찍히는 시대. "MemoryLimiter" 라벨이 답을 알려준다.

개발자 입장에서 일회성 불편처럼 보일 수 있지만, 생태계 전체로 보면 훨씬 공정한 규칙이다. 누수가 없는 앱은 지금까지 이유 없이 피해만 봐왔다. 이제 원인 제공 앱이 책임을 진다.

사용자 입장에서도 체감 이득이 크다.

  • 멀티태스킹이 안정됨 — 백그라운드 앱이 덜 죽음
  • 배터리 소모 감소 — GC·스왑 감소
  • 폼 팩터 전반에서 일관된 경험

5. 지금 당장 해야 할 일

  1. 테스트 환경에 Android 17 Beta 4를 올려라. 실제 문제가 있다면 지금 드러난다.
  2. ApplicationExitInfo 수집을 운영 빌드에 심어라. "MemoryLimiter" 카운트를 대시보드에.
  3. ProfilingManager 기반 이상 힙덤프 자동 수집을 붙여라.
  4. LeakCanary 체크 루프를 PR 리뷰 플로우에 포함시켜라.
  5. 팀의 메모리 예산 베이스라인을 문서화하고 회귀를 CI에서 잡아라.
주의: Beta 4에서는 기준이 보수적이지만, 정식 릴리즈에서 더 조여질 가능성이 높다. 지금 대응하지 않으면 정식 출시 후 사용자 크래시 리포트로 배우게 된다.

6. 마치며

App Memory Limit은 기능 추가가 아니라 규칙 변경이다. Android가 "원하는 만큼 써, 대신 운 나쁘면 누가 죽을지 몰라"에서 "네 몫만큼 써. 넘치면 네가 죽는다"로 바뀌는 것.

이 변화는 당장은 불편할 수 있지만, 모든 앱이 같은 규칙 위에서 경쟁하게 만든다. 좋은 앱은 보상을 받고, 방치된 누수는 숨을 곳이 없어진다. Deterministic Android의 시작점, App Memory Limit을 가볍게 보지 말자.
