Background

왜 모니터링에 벡터 압축이 필요할까?

모니터링 시스템은 매초 수만 개의 숫자를 수집합니다. 이 숫자들을 "벡터"로 묶으면 강력한 분석이 가능해지지만, 저장 비용과 검색 속도가 문제입니다.

핵심 아이디어

서버의 상태를 숫자 묶음(벡터)으로 표현하면, "지금 상태가 정상인지", "과거에 비슷한 상황이 있었는지"를 수학적으로 비교할 수 있습니다.

TurboQuant은 이 벡터를 4~10배 압축하면서도 비교 정확도를 유지하는 기술입니다.

POC 1

이상 탐지: "지금 서버 상태가 정상인가?"

기존 임계치 방식("CPU > 80%면 알림")으로는 복합적인 이상을 잡기 어렵습니다. 벡터 유사도 기반이면 "CPU는 괜찮은데 TPS가 급감하면서 응답시간이 치솟는" 복합 패턴도 탐지할 수 있습니다.

기존 방식과 비교

임계치 방식: "CPU > 80%"만 보면, CPU 70% + TPS 급감 + 응답시간 폭등 같은 복합 이상은 놓침.

벡터 유사도 방식: 7개 메트릭을 동시에 비교하니 "전체적으로 뭔가 이상하다"를 포착할 수 있음.

POC 2

시계열 압축: "메트릭 저장 비용을 줄일 수 있을까?"

서버 1,000대에서 메트릭 100개를 5초마다 수집하면 하루에 17억 데이터포인트. 저장 비용이 만만치 않습니다. 압축하면서도 나중에 분석할 때 정확도가 유지될까요?

실제로 이렇게 측정했습니다

# 6,651개 벡터를 비트별로 압축 & 복원
for b in [1, 2, 3, 4]:
    tq_mse  = TurboQuantMSE(d=7, b=b)
    tq_prod = TurboQuantProd(d=7, b=b)

    for i in range(500):  # 500개 벡터 쌍으로 측정
        # MSE: 원본 vs 복원의 거리
        x_hat = tq_mse.dequant(tq_mse.quant(x))
        mse = sum((x - x_hat)**2)

        # 내적 오차: 원본끼리 내적 vs 압축 후 내적
        true_ip  = dot(x, y)
        approx_ip = tq_prod.inner_product(tq_prod.quant(x), y)
        ip_error = abs(true_ip - approx_ip)

핵심 장점

기존 Product Quantization(PQ)은 새 데이터가 들어오면 전체를 다시 학습해야 합니다 (수백 초). TurboQuant은 data-oblivious라서 새 서버가 추가되든, 메트릭 패턴이 바뀌든 즉시 적용 가능합니다.

모니터링 환경에 딱 맞는 이유

1. 실시간 처리

벡터당 0.01ms. 초당 10만 벡터도 1초 안에 처리. 모니터링 데이터 유입 속도를 충분히 감당합니다.

2. 준비 시간 제로

PQ처럼 k-means 학습이 필요 없습니다. 서버 100대가 갑자기 추가되어도 즉시 압축/검색 가능.

3. 편향 없는 비교

내적(유사도)이 편향되지 않으니 이상 탐지에서 오탐/미탐이 줄어듭니다.

4. 4~16배 저장 절약

같은 스토리지에 4배 더 오래 보관하거나, 스토리지 비용을 75% 줄일 수 있습니다.
