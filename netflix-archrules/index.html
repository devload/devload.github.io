Chapter 0

이게 뭔데?

한 줄 요약

Netflix는 수만 개의 Java 저장소를 관리해. 누가 어떤 코드를 잘못 쓰는지 사람이 일일이 못 보니까, "규칙 위반을 자동으로 잡아주는 로봇 검사관"을 만들어서 모든 저장소에 풀어놓은 거야.

왜 만들었어?

Netflix에는 회사 안에서 만든 Java 저장소(코드 뭉치)가 수만 개나 있어. 한 팀이 만든 라이브러리(공용 부품)를 다른 수백 팀이 가져다 쓰지.

그런데 어느 날 한 라이브러리 작성자가 "이 함수 몇 년 동안 안 쓰는 것 같은데 지워야지!" 하고 코드를 삭제했어. 그랬더니… 아직 그 함수를 쓰고 있던 다른 팀들의 앱이 줄줄이 멈춰버렸어.

비유: 회사 단톡방의 비극

회사에 단톡방이 수만 개 있어. 어떤 팀이 자기 단톡방 공지를 지웠는데, 알고 보니 그 공지를 다른 23개 단톡방이 링크로 걸어놓고 있었어. 한 번에 23개 팀의 업무가 멈춘 거지.

이걸 사전에 막으려면, "누가 내 공지를 링크하고 있는지" 한눈에 보는 시스템이 필요해.

Chapter 1

사고는 왜 났을까?

"몇 년째 안 쓰는 코드"의 함정

라이브러리 작성자 입장에서 생각해봐:

  • "이 함수, 이름이 oldFunction()이야"
  • "몇 년 전에 @Deprecated (곧 지울 거임) 딱지도 붙여놨어"
  • "이제 진짜 안 쓰겠지?"
문제: 확인할 방법이 없어!

수만 개 저장소 중에 누가 내 oldFunction()을 쓰는지 일일이 검색할 수가 없어. Ctrl+F로 회사 전체를 뒤질 수도 없잖아?

그래서 작성자는 "안 쓰겠지~" 하고 지워. 그리고 며칠 뒤 다른 팀들이 비명을 질러. "야, 우리 빌드 망가졌어!"

Chapter 2

먼저 "약속"부터 만들었어

코드에 딱지 붙이기

Netflix는 먼저 라이브러리 작성자들과 약속을 만들었어. 코드에 "이건 어떤 종류야"라고 딱지(어노테이션)를 붙이기로:

딱지 뜻 비유
@Public 마음껏 가져다 써도 돼 도서관 책 — 자유롭게 빌려가
@Experimental 실험 중, 바뀔 수 있음 베타 테스트 — 깨질 수도 있음
@Deprecated 곧 지울 거니까 옮겨가 철거 예정 건물
(딱지 없음) 내부용, 만지지 마 직원 전용 출입구
근데... 약속만으로는 부족해

딱지 붙인다고 사람들이 다 지키나? 안 지키지. 누군가는 "내부용" 출입구로 몰래 들어가고, 누군가는 "곧 철거된다"는 건물에서 계속 살아.

그래서 자동으로 잡아주는 도구가 필요해진 거야.

Chapter 3

ArchUnit이라는 검사관

이미 있던 도구

다행히 세상엔 이런 검사관이 이미 있었어. 이름은 ArchUnit. GitHub에 별 3,500개나 받은 유명 오픈소스 도구야.

ArchUnit이 하는 일

코드에 "이런 규칙은 지켜!"라고 알려주면, ArchUnit이 자동으로 검사해서 위반하는 부분을 찾아줘. 예를 들면:

  • "Guava의 Optional 쓰지 말고 Java의 Optional 써라"
  • "@Deprecated 딱지 붙은 함수는 호출하지 마라"
  • "내부 패키지 함수를 외부에서 부르지 마라"

다른 도구랑 뭐가 다른데? (AST vs Bytecode)

비슷한 검사관으로 PMD라는 게 있어. 근데 둘이 좀 달라:

😵 PMD (옛날 방식)

소스 코드(글자)를 읽어서 검사해.

  • Kotlin이랑 Java는 글자가 다르니까 언어마다 룰 다시 써야 함
  • "숨겨놓은 표현"으로 검사 피해갈 수 있음
  • 커스텀 룰 만들기 진짜 어려움 (XPath 같은 거 써야 함)

😎 ArchUnit (요즘 방식)

컴파일된 바이트코드를 읽어서 검사해.

  • Kotlin이든 Java든 컴파일 후엔 똑같음 → 룰 하나면 끝
  • 최종 실행될 코드를 보니까 못 숨김
  • Java 코드로 룰을 쓸 수 있음 → 자동완성, 테스트 가능
비유: 글짓기 검사 vs 시험지 검사

PMD는 학생이 쓴 글짓기 원본을 보고 검사하는 선생님이야. 영어로 쓴 글, 한국어로 쓴 글, 일본어로 쓴 글마다 다른 선생님이 필요해.

ArchUnit은 글을 다 번역해서 같은 형식으로 만든 다음 검사하는 선생님이야. 한 명이면 모든 언어를 다 검사할 수 있어.

근데 한 가지 문제가...

ArchUnit의 한계

ArchUnit은 한 프로젝트 안에서만 쓰는 도구로 설계됐어. 한 저장소 안에 룰을 적어놓고 거기서만 돌리는 거지.

근데 Netflix는 수만 개 저장소에 동시에 같은 룰을 적용하고 싶어! 어떡하지?

실제 코드는 어떻게 생겼어?

예를 들면, Guava의 Optional을 못 쓰게 막는 룰:

ArchRuleDefinition.priority(Priority.MEDIUM) .noClasses().should() .dependOnClassesThat() .haveFullyQualifiedName("com.google.common.base.Optional") .because("Java Optional is preferred over Guava Optional");

읽어보면 영어 문장처럼 자연스럽지? "어떤 클래스도 → Guava Optional에 의존해서는 안 됨 → 왜냐하면 Java Optional이 더 좋으니까."

처음 문제로 돌아가서

이제 라이브러리 작성자는 매일 아침 자동 리포트를 받아:

  • "네가 deprecated로 표시한 oldFunction()을 아직 쓰는 프로젝트: 23개"
  • "네 내부 API를 몰래 쓰는 프로젝트: 5개"
  • "이 5개만 정리하면 안전하게 지울 수 있어!"

    • → 이제 자신 있게 코드를 정리할 수 있어. "지웠더니 23개 팀이 비명 지름" 같은 사고가 안 남.

    Chapter 6

    앞으로의 계획

    지금까지는: "문제 찾기"

    지금은 "잘못된 코드가 어디 있는지 알려주는 것"까지만 자동화됐어. 고치는 건 아직 사람 몫.

    앞으로는: "자동 수정"

    🔧 OpenRewrite (결정론적)

    "이 패턴은 무조건 이렇게 바꿔" 같은 명확한 규칙으로 자동 수정. 결과가 일정함.

    🤖 LLM (AI, 비결정론적)

    좀 더 복잡한 케이스는 AI가 상황 보고 알아서 고침. 단, 결과가 매번 다를 수 있음.

    최종 그림

    ArchUnit이 "여기, 여기, 여기 문제 있어요"라고 정확하게 짚어주면, 자동 수정 도구가 그 정보를 받아서 코드를 알아서 고쳐. 발견부터 수정까지 사람 손 거의 안 닿는 파이프라인이 되는 거지.

    IDE 통합도 계획 중

    편집기(IDE)에서 코드를 쓰는 순간 빨간 줄로 "이거 규칙 위반이에요"라고 알려주는 기능도 만들 예정이래.

    전체 그림 정리

    📌 5문장 요약

    1. 문제: 수만 저장소에서 deprecated 코드를 누가 쓰는지 추적 불가 → 사고 발생
    2. 약속: @Public / @Experimental / @Deprecated 어노테이션 도입
    3. 도구: 바이트코드 기반 검사 도구 ArchUnit 채택
    4. 확장: 룰을 라이브러리처럼 배포하는 Nebula ArchRules 자체 개발
    5. 결과: 5,000+ 저장소에 358개 룰 자동 적용, 100만 건 문제 탐지

    왜 중요해?

    Netflix 규모(수만 저장소)에서는 사람이 도저히 관리 못 해. 이런 자동화 인프라가 없으면 코드 품질은 시간이 갈수록 무너져. Netflix는 이걸 "맞춤법 검사기"처럼 회사 전체에 깔아둔 거야.

    그리고 더 멋진 건, 각 라이브러리 팀이 자기만의 규칙을 만들어 끼울 수 있다는 점. 중앙에서 다 정해주는 게 아니라, 분산된 작성자들이 각자의 도메인 지식을 룰로 표현하고 자동 적용시키는 거지.

    부록

    용어 사전

    polyrepo
    코드를 한 큰 저장소(monorepo)에 모으지 않고 수많은 작은 저장소로 분리해서 관리하는 전략.
    deprecated
    "곧 사라질 예정"이라고 표시된 코드. 철거 예정 표지판이 붙은 상태.
    어노테이션 (Annotation)
    코드에 붙이는 딱지/태그. @Deprecated처럼 @로 시작.
    AST (Abstract Syntax Tree)
    소스 코드를 나무 형태로 표현한 구조. 글자 단위 분석에 사용.
    바이트코드 (Bytecode)
    Java 코드가 컴파일된 중간 형태. JVM이 직접 실행하는 형식.
    ASM
    Java 바이트코드를 읽고 분석하는 라이브러리. ArchUnit의 엔진.
    Gradle
    Java 프로젝트를 빌드하는 도구. Nebula는 Netflix가 만든 Gradle 확장.
    ServiceLoader
    Java 표준 기능. 구현체를 자동으로 찾아 로드하는 메커니즘. 룰 자동 발견에 사용.
    OpenRewrite
    코드를 규칙대로 자동 변환해주는 오픈소스 도구. 마이그레이션에 사용.
