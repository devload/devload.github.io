Chapter 0

이게 뭔데?

한 줄 요약

구글이 만든 AI용 안드로이드 개발 시험지야. 진짜 깃허브에 올라온 실제 버그 100개를 주고, AI가 얼마나 잘 고치는지 보는 거지. 1등은 72.4%밖에 못 맞췄어!

왜 만들었어?

요즘 ChatGPT, Claude, Gemini 같은 AI가 코딩을 엄청 잘한다고 자랑하잖아? 그런데 안드로이드 앱 개발은 특별해. 그냥 파이썬 코드 짜는 거랑 완전 달라.

비유: 수학 잘한다고 과학도 잘할까?

수학 올림피아드 1등이라고 과학 올림피아드도 1등이 되는 건 아니잖아. 각 과목엔 그 과목만의 특별한 지식이 필요하거든.

AI도 마찬가지야. 일반 코딩 시험(HumanEval 같은 거)은 잘 봐도, 안드로이드 개발은 못 할 수 있어. 그래서 구글이 안드로이드 전용 시험지를 따로 만든 거야.

누가 만들었어?

구글의 안드로이드 개발자 팀이 만들었어. 코드는 전부 깃허브에 공개했고, 누구나 자기 AI를 가져와서 시험을 볼 수 있어.

Chapter 1

기존 AI 시험의 문제

"코딩 잘해요!"라고 하는데...

AI 회사들은 자기네 모델이 "코딩 잘한다"고 자랑해. 근데 보여주는 시험이 대부분 이런 거야:

  • HumanEval: 짧은 파이썬 함수 하나 짜기
  • LeetCode: 알고리즘 문제 풀기
  • MBPP: 한 줄짜리 파이썬 태스크
문제점: 실전과 달라!

이런 시험은 "숙제 잘 푸는 초등학생"을 뽑는 거야. 근데 우리가 궁금한 건 "실제 앱 개발 현장에서 일할 수 있는 직장인"이잖아.

  • 현장의 코드는 엄청 길어 (수천~수만 줄)
  • 파일이 수십 개서로 얽혀있어
  • 안드로이드 특유의 규칙이 있어 (Gradle, AndroidManifest, 버전 호환성...)
  • 이미 만든 테스트를 깨뜨리면 안 돼

안드로이드만의 어려움

비유: 집 짓기 vs 레고 조립

파이썬으로 함수 하나 짜는 건 레고 조립과 비슷해. 블록 몇 개 끼우면 끝.

안드로이드 앱 개발은 진짜 집 짓기야:

  • 벽돌(코드)만 있으면 안 돼 — 도면(Gradle 설정), 배관(권한), 전기(라이프사이클)까지 다 맞춰야 해
  • 집이 다 지어져 있는 상태에서 "2층 화장실만 리모델링" 하는 느낌
  • 리모델링하다가 1층 부엌이 무너지면 큰일 나!

그래서 Android Bench가 등장!

구글은 이렇게 말해:

"우리가 실제로 현장에서 맞닥뜨리는 문제로 시험을 보자.
그래야 어떤 AI가 진짜 쓸만한지 알 수 있지!"

Chapter 2

시험지는 어떻게 만들었어?

진짜 버그를 모았어!

구글 팀은 시험 문제를 지어내지 않고, 진짜 깃허브에 있던 버그들을 가져왔어.

비유: 진짜 사건 수사

경찰 시험을 볼 때, 상상 속 사건을 풀게 하는 게 아니라 실제로 일어났던 미제 사건을 풀게 하는 느낌이야.

이미 사람이 해결한 답(정답지)이 있고, 그 답까지 AI가 도달할 수 있는지 보는 거지.

4단계 고르기

1단계: 깃허브 PR 수집
인기 안드로이드 앱 33개에서 38,989개 Pull Request
2단계: 자동 필터링
⭐500개 이상, 테스트 있음, 이슈 연결됨
3단계: 사람이 검수
컴파일 되나? 난이도 적당한가?
4단계: 최종 100개
안드로이드 전문가가 한 번 더 확인

어떤 문제들이 들었을까?

예시: 진짜 나온 문제 하나

ClockYou 앱 PR #292 — "같은 이름 도시 2개 추가하면 앱이 죽어요"

겉보기엔 간단한 버그 같지?
근데 고치려면 Room 데이터베이스 마이그레이션을 해야 해. 그냥 코드 한 줄 고쳐선 안 되고, 저장된 데이터 구조까지 바꿔야 하는 거야.

이런 "겉은 쉬운데 속은 복잡한" 문제가 진짜 안드로이드 개발의 현실이야.

시험지 구성 자체가 구글의 "이게 좋은 개발자다"는 철학을 담고 있어.

Chapter 3

채점은 어떻게 해?

2단계 채점 시스템

① 추론 단계 (Inference)
AI가 문제를 읽고 코드 패치를 만듦
② 검증 단계 (Verifier)
도커 안에서 패치 적용 → 빌드 → 테스트 실행
점수 계산
모든 테스트 통과해야 1.0 (만점)

0점 아니면 1점! (부분 점수 없음)

중요: 엄격한 2진 채점

이 시험엔 부분 점수가 없어. 90% 맞췄어도 한 테스트라도 실패하면 그 문제는 0점이야.

왜? 실제 현장에선 "거의 됐어요"가 통하지 않거든. 앱이 크래시 나면 사용자는 떠나.

실패하는 5가지 이유

상태 의미 비유
PASSED 패치 적용 + 빌드 + 모든 테스트 통과 만점!
AGENT_FAILED_BUILD 패치 만들었는데 컴파일 실패 답은 썼는데 글씨가 엉망
AGENT_FAILED_TEST 빌드는 성공, 테스트 실패 답 썼는데 틀림
NO_PATCH_GENERATED 패치 자체를 못 만듦 백지 제출
EVAL_ERROR 시스템 오류 시험지 잘못 인쇄됨

최종 점수 = pass@1

공식

점수 = 통과한 문제 수 ÷ 100

pass@1이란? — "한 번 시도해서 맞췄는가"라는 뜻이야.

여러 번 시도해서 한 번이라도 맞추기 X. 첫 시도에 맞춰야 해. 실제 개발자처럼!

공정함을 위한 장치들

왜 이렇게까지?
  • 도커 샌드박스: 모든 AI가 똑같은 환경에서 시험 봄
  • 10번 반복: AI마다 5~10번씩 시험 봐서 평균 냄 (운빨 제거)
  • 95% 신뢰구간: 부트스트랩으로 1000번 돌려서 점수 범위 제시
  • 카나리 문자열: 미래 AI가 이 시험을 미리 외우는 부정행위 방지
  • 회귀 테스트: 버그 고치다가 다른 기능 망가뜨리면 0점!

🔥 가장 중요한 숨은 룰

pass_to_pass

각 문제엔 두 가지 테스트가 있어:

  • fail_to_pass: 버그 때문에 실패하던 테스트 → 이제 통과해야 함
  • pass_to_pass: 원래도 잘 돌던 테스트 → 계속 잘 돌아야 함

즉, "내 버그 고치면서 남의 멀쩡한 코드 건드리지 마라"가 점수로 강제되는 거야!

이 결과가 말하는 것

1) "만능 AI"는 아직 없어

1등도 72%라고?

이게 무슨 말이냐면, 1등 AI도 100문제 중 28개는 틀린다는 거야.

즉, "AI한테 다 맡기면 안드로이드 앱 나온다"는 아직 환상이야. 사람 리뷰가 꼭 필요해.

2) 공동 1등 — 흥미로운 사실

자기네 플랫폼에서 자기가 1등 못 했네?

재밌는 건 구글이 만든 시험인데 구글 Gemini만 1등을 한 게 아니야. OpenAI의 GPT-5.4와 공동 1등이야.

이게 바로 이 시험의 신뢰도를 높여. 구글이 자기네 모델 편들려고 만든 게 아니라는 증거거든.

3) 경량 모델은 처참

Flash 모델들, 16~42%

Gemini 2.5 Flash는 16%밖에 못 맞췄어. 이건 거의 못 하는 수준.

교훈: 비용 아끼려고 가벼운 AI 쓰면 안드로이드 개발엔 적합하지 않아.

4) 최상위 3개 AI의 차이

1등(72.4%)과 4등(66.6%)의 차이는 약 6%.
실전에선 이 정도 차이가 커. 100개 버그 중 6개를 더 고친다는 뜻이니까.

Chapter 5

구글이 생각하는 "좋은 안드로이드 코드"란?

시험지는 그냥 "기능이 되는지"만 보는 것 같지? 그런데 자세히 보면, 이 시험은 구글이 생각하는 "좋은 코드"의 기준을 숨기고 있어.

구글의 답

"혼자 잘 짜는 천재"가 아니라, 팀 코드베이스에 잘 녹아드는 실용가가 좋은 안드로이드 개발자다!

5가지 숨은 기준

1) 모던 안드로이드 스택(MAD) 숙련도

구글이 미는 기술들
  • UI: Jetpack Compose
  • 언어: Kotlin
  • 비동기: Coroutines + Flow
  • DB: Room
  • 의존성 주입: Hilt
  • 네비게이션: Navigation Compose
  • 빌드: Gradle (KTS)

이 기술들을 못 쓰는 AI는 시험에서 계속 떨어져. 구글은 "이게 표준이야"라고 공식적으로 선언한 거지.

2) 컨벤션 동화력

비유: 전학생의 매너

전학 간 첫날, 너 혼자만 반 전체랑 다른 옷을 입고 오면 어떨까? 아무리 옷이 비싸도 이상해 보이잖아.

코드도 똑같아. 기존 코드가 스네이크 케이스를 쓰는데 너 혼자 카멜 케이스 쓰면 이상하지.

시험 프롬프트에 이렇게 쓰여 있어: "general and consistent with the codebase (코드베이스와 일관되게)".

3) 회귀 없는 최소 침습 수정

비유: 수술의 기본

맹장 수술하러 갔는데 의사가 "어차피 열었으니까 간도 좀 수술할까요?" 하면 무섭지?

코드 수정도 그래야 해. 딱 필요한 부분만 고쳐야 하고, 주변 기능을 부수면 안 돼.

그래서 pass_to_pass 테스트가 존재하는 거야. 멀쩡하던 기능을 건드리면 바로 0점.

4) 플랫폼 진화 추적

안드로이드는 매년 바뀌어

안드로이드는 매년 새 버전이 나와. API가 deprecated 되고, 새 권한이 생기고, SDK 타겟이 올라가.

시험엔 이런 문제들이 많아:

  • "안드로이드 14에서 새 권한 요구사항이 생겼어. 대응해줘"
  • "foldable 폰에서 configuration change 처리"
  • "Navigation 2에서 3으로 마이그레이션"

옛날 지식만으론 못 풀어. 최신 트렌드를 계속 따라잡는 능력이 필요해.

5) 멀티 파일 의존성 이해

비유: 거미줄 끝 당기기

거미줄 한 가닥을 당기면 연결된 모든 줄이 떨림이 있지?

안드로이드 코드도 그래. MainActivity.kt 하나 고치려면:

  • 그 파일이 쓰는 ViewModel도 봐야 하고
  • ViewModel이 쓰는 Repository도 봐야 하고
  • Repository가 쓰는 Database Entity도 봐야 하고
  • 관련 테스트 파일도 다 봐야 해

AI가 이 연결고리를 이해해야 점수를 받을 수 있어.

한 문장으로 요약하면

Google's Quality Definition

"현실 코드베이스에 자연스럽게 녹아드는, Modern Android Development 스택 기반의, 최소 침습적이고 회귀 없는 변경."

— 아키텍처 순수성보다 실용적 역량을 품질의 척도로 본다!

Chapter 6

그래서 뭐가 중요해?

우리가 얻는 교훈

🧑‍💻 개발자라면

  • AI에게 안드로이드 코딩 맡길 땐 Opus / GPT-5 / Gemini 3 Pro급 모델 쓰기
  • Flash 같은 경량 모델은 자동완성 용도로만
  • AI가 만든 패치는 반드시 기존 테스트 다시 돌려보기
  • 구글이 공식으로 미는 MAD 스택 익히기 (Compose + Kotlin + Coroutines + Hilt + Room)
  • "기능만 되면 되지"가 아니라 "주변 기능 안 망가지게"를 항상 고려

🏢 팀 리더라면

  • AI 코딩 도구를 선택할 때 일반 벤치마크 말고 Android Bench 점수 참고
  • 사내 코드베이스에도 이런 시험을 만들어두면 좋음 (fail_to_pass / pass_to_pass 개념)
  • AI 도입 = 개발자 대체 ❌ → AI 결과를 리뷰하는 리뷰어 스킬이 더 중요해짐

🎓 AI 연구자라면

  • 1등도 72%라는 건 아직 연구할 게 많다는 뜻
  • 특히 어려운 태스크: 멀티 파일 수정, Gradle 설정, 스크린샷 이해 필요한 UI 버그
  • 오픈소스 데이터셋 공개 → 파인튜닝 벤치마크로도 활용 가능

정리: 한 장으로 이 시험

Android Bench 요약 카드
  • 누가: 구글 안드로이드 팀
  • 왜: 일반 코딩 벤치로는 안드로이드 실력 못 재
  • 뭘: 실제 깃허브 PR 100개 (Kotlin/Compose 위주)
  • 어떻게: AI가 패치 만들면 빌드+테스트로 자동 채점
  • 점수: pass@1 (한 번에 맞혀야 함, 부분점수 X)
  • 결과: 1등 72%, 경량 모델 16% — 격차 큼
  • 의미: "좋은 안드로이드 개발 = 팀 코드에 잘 녹아드는 최소 침습 수정"

마지막 한마디

Android Bench는 그냥 "어떤 AI가 잘하나" 줄 세우는 게 아니야.
"좋은 안드로이드 코드란 무엇인가"라는 질문에 구글이 공식적으로 내놓은 답이기도 해.

그리고 그 답은: "혼자 잘 짜는 천재가 아니라, 팀에 잘 녹아드는 실용가".
AI도, 사람도, 결국 같은 기준이야!
